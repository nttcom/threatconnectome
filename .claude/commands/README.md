# EOL Product Addition Command Guide

Claude Code slash commands for adding new products to the EOL (End-of-Life) feature in Threatconnectome.

## Workflow Overview

```
/eol-sbom [product]
    ↓  Generate SBOM and register it in TC
/eol-investigate [product]
    ↓  Output investigation report to eol-work/
(Review report and edit if needed)
    ↓
/eol-implement [product]
    ↓  Implement matching logic and run tests
Run scripts/endoflife2tc.py
    ↓  Register EoL data in TC
Verify in TC UI
```

## Prerequisites

- Docker and trivy must be installed
- Threatconnectome must be running locally (`http://localhost/api/docs` must be accessible)
- Obtain pteam_id and access_token in advance

## Command Reference

### `/eol-sbom [product]`

Creates Dockerfiles for each ecosystem (rocky / alpine / ubuntu / pypi / golang / npm), generates SBOMs with trivy, and registers them in TC.

**Argument**: Product name on `endoflife.date` (e.g., `nginx`, `django`, `apache-http-server`)

**Main steps**:
1. Determine which ecosystems the product applies to
2. For each ecosystem: build Docker image → generate SBOM → upload to TC
3. Remove used images

**Base images used** (defaults to a recently EOL-reached version to maximize EOL matches):

| ecosystem | base image |
|-----------|-----------|
| rocky     | rockylinux:8 |
| alpine    | alpine:3.18 |
| ubuntu    | ubuntu:20.04 |
| pypi      | python:3.9-slim |
| golang    | golang:1.21 |
| npm       | node:18 |

**Output**: `eol-work/trivy-<product>-<ecosystem>.json`

---

### `/eol-investigate [product]`

Run after `/eol-sbom`. Makes no code changes.

**Argument**: Product name on `endoflife.date`

**Main steps**:
1. Fetch release info and version format from the endoflife.date API
2. Query TC DB to check package names and versions per ecosystem
3. Determine `is_ecosystem` / `product_category` / matching class
4. Output investigation report

**Output**: `eol-work/<product>-report.md`

---

### `/eol-implement [product]`

Run after `/eol-investigate`. Review the report before proceeding.

**Argument**: Product name (same as used with eol-investigate)

**What gets implemented**:

| is_ecosystem | Files changed |
|---|---|
| False (Package type) | `scripts/endoflife2tc.py`, `api/app/business/eol/product/<product_snake>_product.py` (new), `eol_product_factory.py`, `eol_version_factory.py` |
| True (Ecosystem type) | `scripts/endoflife2tc.py`, `api/app/business/eol/ecosystem/eol_<product_snake>_ecosystem.py` (new), `eol_ecosystem_factory.py` |

`$product` は Claude コマンド引数 (endoflife.date のプロダクト名) です。`<product_snake>` は `$product` を Python モジュール名として有効な snake_case に変換した派生値です。
例: `apache-http-server` -> `apache_http_server`

After implementation, black / ruff / mypy / codespell and pytest are run automatically.

---

## Usage Example

```
/eol-sbom nginx
/eol-investigate nginx
(Review eol-work/nginx-report.md)
/eol-implement nginx
```

## Working Directory `eol-work/`

A temporary directory for intermediate files generated by each command.

| File | Generated by | Content |
|---|---|---|
| `trivy-<product>-<ecosystem>.json` | eol-sbom | CycloneDX format SBOM |
| `<product>-eol.json` | eol-investigate | endoflife.date API response |
| `<product>-report.md` | eol-investigate | Investigation report |

> **Note**: Do not commit `eol-work/` to the repository.

## FAQ

**Q. EOL matching is not triggered**
- If the product version is still supported on endoflife.date (eolFrom is null), no matching occurs. This is expected behavior. Example: Apache HTTP Server 2.4 is an active branch, so packages with version 2.4.x will not match.
- OS-level EOL matching (e.g., the ubuntu-20.04 ecosystem itself) does occur.

**Q. Changes are not reflected in TC after running `eol-implement`**
- Run `scripts/endoflife2tc.py` manually. This script registers EOL version data in TC.

**Q. Not sure which ecosystems to target**
- `/eol-sbom` determines this automatically. If uncertain, check the identifiers on endoflife.date or the package repositories for each ecosystem.
